In this blog post, I will be discussing how I broke into my ISP router by using a command injection, how I found another even better command injection inside, and how I ended up finding and exploiting a remote heap overflow in the kernel driver of the Mediatek chipset inside the router. These vulnerabilities have been disclosed over the past months and have been assigned the following CVEs: - CVE-2025-13942: Zyxel unauthenticated command injection in UPnP daemon - CVE-2025-13943: Zyxel authenticated command injection in log export CGI - CVE-2026-XXXX: Mediatek MTXXX WPS heap overflow As usual, this research is heavily based on reverse engineering and a fair amount of trial and error, so excuse me if there are any mistakes, specially in low level kernel stuff, since this is a very complex topic. If you find something that is awfully wrong and want to help me correct it, I am absolutely open to it. Anyways, let's start with the blog, since there is a lot to cover. [ Some context ] This journey begins in late summer 2025, when my colleague Sergio and I decided to give Pwn2Own Ireland 2025 a shot, targeting a couple of embedded devices. However, we underestimated the tight timeframe of the competition, and our day jobs did not leave us with enough time to properly focus on it. After many intense days (and nights) of hacking, we were not able to come up with a working exploit (although we did collect a decent number of DoS scripts lol). After dealing with the frustration, I decided to take a break from hacking for a few months and dedicate my free time to resting and enjoying life. Of course, that did not happen and, instead, I somehow ended up diving into this massive research project. The thing is that I had changed my Internet provider a couple of months ago, so my new ISP had delivered a new router. Although I am obviously not going to refer to this ISP by its name because it probably won't never fix anything described in here, it is a medium sized one here in Spain that was gaining a lot of popularity at the moment due to offering good quality Internet at a low price. So, as always, I wanted to know what was inside this new router that would effectively act as the door to my home. The router was a Zyxel device, but I did not know the model yet. I noticed that it supported both SSH and telnet, so I connected using the credentials that the ISP guy gave me when installing the router ("user:user" lol) and was presented with a restricted shell: $ ssh -oHostKeyAlgorithms=+ssh-dss -oPubkeyAcceptedAlgorithms=+ssh-dss user@192.168.1.1 user@192.168.1.1's password: No entry for terminal type "xterm-256color"; using dumb terminal settings. I ended up finding an article, which I won't be sharing because it explicitely names the ISP, that described a way of extracting the admin password from the web interface. It basically exploited an authenticated command injection in the dynamic DNS configuration page. I tried this technique, but nothing seemed to happen. Also, when introducing special characters in other forms to check if there was any other easy command injections, the following message appeared ("It is not possible to write special characters"), and no request was sent: Further analysis revealed that authenticated HTTP requests had their bodies encrypted, which was enough to keep me away for the moment. I stopped there and did not continue further. [ Breaking into the router ] However, as usual, I could not stop thinking about this, since an ISP router seemed a fairly easy target and was starting to damage my ego. So, I just created a new folder for this project in my vuln research directory and started diving deeper into this. First thing first, I did a proper port scan, since I had not executed one because I was only taking a look earlier: $ nmap -sV -O -Pn -p- -oN nmap.log 192.168.1.1 Nmap scan report for _gateway (192.168.1.1) Host is up (0.025s latency). Not shown: 65526 closed tcp ports (reset) PORT STATE SERVICE VERSION 21/tcp open ftp 22/tcp filtered ssh 23/tcp filtered telnet 53/tcp open domain (unknown banner: Domain Name Server) 80/tcp open http 443/tcp open ssl/https 2601/tcp open zebra? 49152/tcp open unknown 49153/tcp open unknown [...] The ftp server allowed login as "user", but only allowed access to the "user" home folder, which was empty. Regarding the other ports, "zebra" corresponds to a deprecated routing protocol and ports 49152 and 49153 WiFi configuration UPnP services (remember these, since they will be heavily discussed later). So, given I had no way in, I decided to download the regular firmware for "Zyxel EX3301-T0" routers from the vendor page and compare it to my ISP router. This way, I could check if they were similar enough to try finding something in the default firmware that affected my router and could allow me in. I knew for sure that my ISP had modified it a bit, since some logos, colors and a lot of logs from the UART output confirmed it, but I believed that core functionalities should remain the same. I downloaded the firmware and extracted it using "binwalk", which easily gave me a squashfs filesystem: $ binwalk V550ABVY6.2C0.bin DECIMAL HEXADECIMAL DESCRIPTION -------------------------------------------------------------------------------- 372 0x174 LZMA compressed data, properties: 0x6C, dictionary size: 8388608 bytes, uncompressed size: 16279488 bytes 4180222 0x3FC8FE Squashfs filesystem, little endian, version 4.0, compression:lzma, size: 26706310 bytes, 2393 inodes, blocksize: 262144 bytes, created: 2025-04-11 08:20:33 My main target were the UPnP ports that appeared in the nmap scan, since UPnP services tend to be straightforward and are easily reversed. After some initial analysis, I could not find any binaries that used those ports nor the schema advertised by the service, so I assumed it was an ISP thing and moved on. By then, I just wanted a way inside the router, so I did not care about the potential findings requiring authentication.
