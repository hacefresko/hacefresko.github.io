In this blog post, I will be discussing how I broke into my ISP router by using a command injection, how I found another even better command injection inside, and how I ended up finding and exploiting a remote heap overflow in the kernel driver of the Mediatek chipset inside the router. These vulnerabilities have been disclosed over the past months and have been assigned the following CVEs: - CVE-2025-13942: Zyxel unauthenticated command injection in UPnP daemon - CVE-2025-13943: Zyxel authenticated command injection in log export CGI - CVE-2026-XXXX: Mediatek MTXXX WPS heap overflow As usual, this research is heavily based on reverse engineering and a fair amount of trial and error, so excuse me if there are any mistakes, specially in low level kernel stuff, since this is a very complex topic. If you find something that is awfully wrong and want to help me correct it, I am absolutely open to it. Anyways, let's start with the blog, since there is a lot to cover. [ Some context ] This journey begins in late summer 2025, when my colleague Sergio and I decided to give Pwn2Own Ireland 2025 a shot, targeting a couple of embedded devices. However, we underestimated the tight timeframe of the competition, and our day jobs did not leave us with enough time to properly focus on it. After many intense days (and nights) of hacking, we were not able to come up with a working exploit (although we did collect a decent number of DoS scripts lol). After dealing with the frustration, I decided to take a break from hacking for a few months and dedicate my free time to resting and enjoying life. Of course, that did not happen and, instead, I somehow ended up diving into this massive research project. The thing is that I had changed my Internet provider a couple of months ago, so my new ISP had delivered a new router. Although I am obviously not going to refer to this ISP by its name because it probably won't never fix anything described in here, it is a medium sized one here in Spain that was gaining a lot of popularity at the moment due to offering good quality Internet at a low price. Regarding the weirder ports, "zebra" corresponded to a deprecated routing protocol and ports 38400, 49152 and 49153 were UPnP services. After querying each of them and doing some research, I identified the following:
  1. Port 38400: Service "InternetGatewayDevice", which uses a standardized OCF Device Control Protocol that allows LAN clients to manage NAT port mappings, retrieve the external WAN IP and control firewall rules.
  2. Ports 49152/49153: Service "WFADevice", which uses a standardized Wi-Fi Alliance Device Control Protocol that allows clients to configure 802.11 parameters (SSID, PSK, security settings) over an out-of-band IP channel using the WPS Simple Config protocol.
However, none of these offered direct access to the OS. So, given I had no clear way in, I decided to download the regular firmware for "Zyxel EX3301-T0" routers from the vendor page and compare it to my ISP router. This way, I could check if they were similar enough to find something in the default firmware that also affected my device. I knew for sure that my ISP had modified it a bit, since some logos, colors and a lot of logs from the UART output confirmed it, but I believed core functionalities would remain the same. Regarding ports 49152 and 49153, the binary behind them was "/usr/bin/wscd", but I it only implemented 4 UPnP commands, of which only 3 accepted user input. Then, I found that all commands simply forwarded user input to kernel space using "ioctl", which seemed quite complicated at the moment (keep this in mind, as it will be relevant later). By then, I just wanted a quick way inside the router, so I did not care about the potential findings requiring authentication, so I decided to take a look into the web interface, since I already knew that there had been command injections found there in the past. I quickly found the executable "/bin/zhttpd", which looked like the one behind the web server. I opened Ghidra and started reversing it. For this research, I used Claude a lot, since it can be connected to Ghidra using Cluade desktop and GhidraMCP. Athough Claude still has many flaws when diving deep into reversing and exploitation, it is pretty much useful in the initial phase, in which the main goal is just figuring out which are the most interesting functions of an executable, tracing user input, indentifying global tables, figuring out structures, etc. Using this setup, I quickly identified a global table starting at that mapped routes with their handler functions: This allowed me confirming that the "/bin/zhttpd" binary that I had was close enough with the one in my ISP router, since most of these routes were accessible. After a quick look at each of them, "/cgi-bin/LogResult?action" caught my atention, which was mapped to function "zHttpLogHandle". This endpoint was responsible for managing device log operations and supported multiple actions through the "action" parameter, such as creating network traffic captures ("START_CAPTUR", "STOP_CAPTUR"), deleting logs ("DEL_LOG"), exporting logs ("EXPORT_LOG"), etc. In order to manage network traffic captures, it used system commands to execute and then stop "tcpdump": When executing the log export functionality, function "copyLog" was called, which reads a previously generated "tcpdump" capture file and writes its contents into a new log file. The problem here is that "copyLog" processes the network capture line by line using "fgets" and then, for each line read, it constructs a shell command using a "snprintf" wrapper. Then, it directly executes it using "system": Since this process does not include any sanitization of the binary data that comes out of the network capture, any string that containing shell metacharacters will be escapes and later executed, enabling remote command execution. Perfect. Now, in order to exploit this, the attacker had to be authenticated. Authenticate to the web interface with valid credentials to obtain a session key. 2. Start a network traffic capture 3. Send an HTTP request to the router with a header containing a command injection payload. 4. Stop the network traffic capture 5. However, the one that caught my attention was TCP port 38400, since its corresponding binary, and therefore the one exposing the "InternetGatewayDevice" UPnP service, was "zupnp", which was nowhere to be found in the stock EX3301-T0 firmware. This meant it was either an ISP-developed binary or part of some Zyxel SDK made available to OEMs. Anyways, the fact that it wasn't in the public firmware made it an interesting target, so I loaded it into Ghidra and started reversing it. Out of all UPnP commands in the "InternetGatewayDevice" service, "AddPortMapping" was the most interesting one, since it accepted a lot of user input in order to create a new NAT port forwarding rule. Upon receiving a request, function "zupnpPortMappingExecute" was eventually executed, which then called another function at address "0x0040f390" that processed the supplied parameters and added the corresponding NAT rule in the system firewall using "iptables". However, the function did not implement any user input validation, which got inserted directly into the "iptables" command that got executed using "zsystem", a "system" wrapper: This obviously constituted another command injection vulnerability, this time in an unauthenticated service. Exploitation was a bit trickier, though. The parameters susceptible to injection, "NewRemoteHost" and "NewInternalClient", were copied into fixed-size local buffers using "strncpy" before being inserted into the command string, limiting each of them to 15 characters: This meant that a single request was not enough to inject a meaningful payload. To work around this, I created an exploit that builds a temporary script on the device character by character across multiple requests and then executes it. Since both "NewRemoteHost" and "NewInternalClient" are inserted into the same "iptables" command at different positions, both fields can be injected simultaneously on each request. "NewRemoteHost" is used to define a shell variable containing the path of the temporary script with "a=/tmp/a", while "NewInternalClient" is used to append one character at a time to it with "printf X>>$a", where "X" is the character being written. The full command executed on each request is the following: iptables -t nat -A NAT_PORT_MAPPING_UPNP_1 -p tcp --source ;a=/tmp/a;" --dport 1337 -j DNAT --to ";printf X>>$a;:1337 The command gets divided into the following separate commands: iptables -t nat -A NAT_PORT_MAPPING_UPNP_1 -p tcp --source ; a=/tmp/a; # <-- NewRemoteHost " --dport 1337 -j DNAT --to "; printf X>>$a; # <-- NewInternalClient :1337 Note that the quotes are intentionally injected as part of the exploit. Since the shell would otherwise stop executing after the first injected command, the closing quote from "NewRemoteHost" and the opening quote from "NewInternalClient" are used to keep the rest of the "iptables" command syntactically valid, ensuring all injected commands are executed. Once the full reverse shell command is assembled in "/tmp/a", a final request executes it with ";sh /tmp/a;", achieving unauthenticated Remote Code Execution as root :D Again, this vulnerability was reported to Zyxel and assigned CVE-2025-13942.
